## 一、完善全局错误处理

前面几章，错误处理散落在各处（每个 `try/catch` 里各自 `ElMessage.error`）。其实可以在 **axios 拦截器里统一处理**——所有请求出错，自动弹错误提示，组件里就不用重复写了。

更新 `src/api/request.js`：

```js
import axios from 'axios'
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: '/api',
  timeout: 10000
})

request.interceptors.response.use(
  (response) => response.data,
  (error) => {
    // 全局错误提示：所有请求失败都会自动弹
    const msg = error.response?.data?.message || error.message || '请求失败'
    ElMessage.error(msg)
    return Promise.reject(error)
  }
)

export default request
```

这样，**任何一个接口请求失败，都会自动弹一个错误提示**，组件里只需处理成功逻辑（或捕获后做额外处理）。这就是"拦截器统一处理"的价值——对照 jQuery，你以前得在每个 `$.ajax` 的 `error` 回调里重复写错误提示。

> 💡 同理，`loading` 也可以全局化（用一个 store 或拦截器配合，请求时自动显示全屏 loading）。本教程保持每个页面自己控制 `loading`，更直观；真实项目常做全局 loading。

## 二、打包：`npm run build`

开发时用 `npm run dev`（带热更新，但体积大、未优化）。**上线前要打包**——压缩、合并、优化代码：

```bash
npm run build
```

执行完，项目里会多出一个 **`dist/`** 目录。打开看看：

```
dist/
├── index.html
└── assets/
    ├── index-abc123.js     ← 所有 JS 压缩合并，带哈希（缓存用）
    ├── index-def456.css    ← 所有 CSS 压缩合并
    └── ...                  ← 图片等资源
```

这就是**上线的全部内容**——一个纯静态的 `dist/` 目录，里面是 HTML + JS + CSS。**没有任何 Vue 源码、没有 node_modules**，体积小得多。

| 阶段 | 命令 | 产物 |
|---|---|---|
| 开发 | `npm run dev` | 内存中、带热更新、未压缩 |
| 上线 | `npm run build` | `dist/` 纯静态、压缩优化 |

## 三、预览产物：`npm run preview`

打包完，想本地看看 `dist/` 长啥样、能不能跑：

```bash
npm run preview
```

它会用一个小型静态服务器跑起 `dist/`，打开终端给的地址（通常 `http://localhost:4173`）就能看到打包后的效果。

> 注意：预览时**没有** `npm run server`（json-server）的话，接口请求会失败——因为后端只在开发时用。预览主要看页面结构和打包是否正常。

## 四、部署上线

`dist/` 是纯静态文件，**放到任何能托管静态文件的地方都能跑**：

| 方式 | 说明 |
|---|---|
| **Nginx** | 把 `dist/` 内容扔到服务器，Nginx 指向它（最常见） |
| **Vercel / Netlify** | 一键部署，连仓库就自动打包上线（最省心） |
| **GitHub Pages / Gitee Pages** | 免费静态托管 |
| **对象存储**（OSS/COS） | 把 `dist/` 上传到云存储，开静态网站托管 |

**对照 jQuery**：jQuery 项目上线，也是把 html/js/css 上传到服务器（以前你可能用 FTP）。Vue 一样，只是多了一步 `npm run build` 打包。

⚠️ **别忘了后端**：本教程的后端是 `json-server`（仅开发用，不上线）。真实上线时，你要有一个**真正的后端服务**（Node/Java/Python...），然后把前端 `request.js` 的 `baseURL` 指向它（比如 `'https://api.yoursite.com'`），而不是开发时的 `/api`。

## 五、history 路由的 404 问题

第 9 章我们用了 `createWebHistory`（URL 干净，如 `/users/3`）。但部署后，**直接访问或刷新 `/users/3` 可能 404**——因为服务器上根本没有 `/users/3` 这个文件，它只是前端的"假路由"。

**解决办法（任选其一）：**

**方案 A：服务器配置 fallback（推荐）**

让服务器把所有找不到的路径都回退到 `index.html`，交给前端路由处理。以 Nginx 为例：

```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

**方案 B：改用 hash 模式**

把 `router/index.js` 里的 `createWebHistory()` 换成 `createWebHashHistory()`，URL 会变成 `/#/users/3`（多个 `#`）。好处是**不用配服务器**，坏处是 URL 不好看。

> 入门阶段用 history + 服务器 fallback 最正规。本地开发（`npm run dev`）不会有这问题，Vite 自动处理了。

## 六、回顾：你完成了什么 🎉

走到这里，你已经从一个 jQuery 程序员，**亲手搭出了一个完整的 Vue3 后台系统**。回头看看你掌握的：

| 能力 | 对照 jQuery |
|---|---|
| 响应式（`ref`/`computed`） | 告别 `.text()`/`.val()` 手动改 DOM |
| 模板指令（`v-if`/`v-for`/`v-model`...） | 告别 `show/hide/each/val` + 拼 HTML |
| 组件化（`props`/`emit`/`slot`） | 告别复制粘贴 5 份 |
| `<script setup>` + Vite | 现代工程化，告别一个 html 写到底 |
| axios + json-server | 现代版 `$.ajax`，前后端分离 |
| Element Plus | 现成表格/表单/弹窗，告别手写 CSS |
| Vue Router | SPA 多页面，告别整页刷新 |
| Pinia | 全局状态，告别全局变量/props 钻孔 |

最重要的，你完成了**一次根本性的思维转变**：

> **从"操作 DOM"到"操作数据"。**

这条转变一旦完成，你以后学 React、Solid、Svelte，都是同一套思想——声明式 + 组件化 + 数据驱动。这是一辈子受用的底层能力。
