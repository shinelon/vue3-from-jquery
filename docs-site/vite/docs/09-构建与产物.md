# 第 9 章 · 构建与产物：vite build 到底做了什么

开发用 `npm run dev`，上线用 `npm run build`。这一章拆解 `vite build` 究竟做了什么、产物长什么样、怎么优化它。

> 📂 配套项目 `project/vite-notes/`，本章对应分支 `vite/06-build`。

---

## 一、`vite build` 做了什么

一句概括：**把你的源码，编译、压缩、优化成一堆"浏览器能直接跑、体积尽量小"的静态文件，放进 `dist/`。**

具体几步：

| 步骤 | 干什么 |
|---|---|
| 编译 | `.vue` → JS，`.ts` → JS，新语法 → 兼容语法 |
| Tree-shaking | 删掉没被用到的代码（你 import 了某库的一小部分，不会把整个库打进来） |
| 压缩 | 去空格、删注释、缩短变量名（JS 用 esbuild，CSS 也压缩） |
| Chunk 拆分 | 把代码切成多个文件（按需加载、缓存友好） |
| 加 hash | 文件名加指纹（`index-D9Jz6BI5.js`），内容变 hash 就变，防浏览器旧缓存 |
| 输出 | 全部写到 `dist/` |

> 📌 关于打包器：Vite 5 用 **Rollup** 打包；**Vite 8 起默认用 Rust 重写的 Rolldown**（更快、能处理更大项目）。对你写配置而言，`build.rollupOptions` 这套 API 依然兼容，不用操心。

---

## 二、看看产物长啥样

在 `vite/06-build` 分支跑 `npm run build`，生成 `dist/`：

```
dist/
├── index.html               ← 入口 html（引用了打包后的 js/css）
├── assets/
│   ├── index-zSEvVU33.css   ← 所有 CSS 合并 + 压缩 + hash
│   ├── index-D9Jz6BI5.js    ← 所有 JS 合并 + 压缩 + hash
│   └── logo-xxxx.svg        ← 你 import 的图片（带 hash）
└── robots.txt               ← public/ 里的，原样拷贝
```

终端会报告每个文件的体积和 gzip 后大小：

```
dist/index.html                   0.42 kB │ gzip:  0.31 kB
dist/assets/index-zSEvVU33.css    0.25 kB │ gzip:  0.20 kB
dist/assets/index-D9Jz6BI5.js   107.35 kB │ gzip: 41.37 kB
```

> 💡 **看 gzip 体积**，那才是用户实际下载的大小。107KB 的 JS，gzip 后才 41KB。

---

## 三、默认的问题：全打成一个 chunk

注意上面：所有 JS（vue + axios + 你的代码）**全塞进一个 `index-xxx.js`**（107KB）。问题：
- 你只改了一行业务代码，**整个 107KB 文件 hash 全变**，用户要重新下载全部（包括没变的 vue）；
- vue 这种"万年不变"的库，本该单独拆出来让浏览器长期缓存。

---

## 四、优化：用 `manualChunks` 拆分

把"几乎不变的依赖"（vue、axios）单独拆成 chunk，业务代码另成一个。这样改业务代码，只有业务 chunk 的 hash 变，vue chunk 缓存命中。

在 `vite.config.js` 加 `build.rollupOptions.output.manualChunks`，用**函数形式**：

```js
export default defineConfig({
  // ... plugins / resolve / server 同前
  build: {
    target: 'es2015',                 // 编译目标：兼容到 ES2015 的浏览器
    rollupOptions: {
      output: {
        // 函数形式：根据模块路径返回 chunk 名，决定它归到哪个 chunk
        manualChunks(id) {
          if (id.includes('node_modules/vue') || id.includes('node_modules/@vue')) return 'vue'
          if (id.includes('node_modules/axios')) return 'axios'
          // 不返回的（你的业务代码）归入默认 chunk
        }
      }
    }
  }
})
```

> ⚠️ **Vite 8（Rolldown）必看**：`manualChunks` 必须用**函数形式**。老教程常见的对象形式 `{ vue: ['vue'] }` 在 Rolldown 下会报 `manualChunks is not a function`——这是 Vite 8 和旧版的一个真实差异。

再 build，产物拆成了多个 chunk（vite-notes 实测）：

```
dist/assets/vue-6SHBIMpL.js               60.98 kB │ gzip: 24.09 kB   ← vue，几乎不变，长期缓存
dist/assets/axios-CMNqsMR2.js             48.97 kB │ gzip: 18.31 kB   ← axios
dist/assets/index-AN4Fh_me.js              2.20 kB │ gzip:  1.18 kB   ← 你的业务代码（从 107KB 降到 2KB！）
dist/assets/rolldown-runtime-DK3Fl9T5.js   0.15 kB │ gzip:  0.15 kB   ← Rolldown 运行时（Vite 8 特有，很小）
```

> 🔑 收益：你改业务代码重新上线，用户只需重新下载 `index-xxxx.js`，`vue-xxxx.js` 从浏览器缓存直接取——加载快、省流量。

## 五、`base`：部署在子路径时要注意

如果你的站点不在根路径（比如部署在 `https://example.com/notes/` 而非根），所有资源 URL 会 404。这时要配 `base`：

```js
export default defineConfig({
  base: '/notes/',    // 所有资源 URL 前加这个前缀
  // ...
})
```

> ⚠️ 这是上线最常见的坑之一，下一篇（部署）会重点讲。

## 六、产物分析：看体积都花在哪（可选）

想知道"107KB 都被什么占了"？装个可视化工具：

```bash
npm install -D rollup-plugin-visualizer
```

配进 plugins，build 后会生成一张交互图（`stats.html`），每个模块的体积一目了然，专治"打包怎么这么大"。

> 💡 这是**可选**的优化工具，本教程不强装。知道有这东西，项目大了用得上。

---

## 对照：上线准备这件事

| | 裸 HTML 时代 | Vite build |
|---|---|---|
| 上线文件 | 你写的源 html/js（没优化） | `dist/` 里压缩、tree-shake、hash 过的文件 |
| 体积 | 大（没删没用的、没压缩） | 小（gzip 后通常小一个数量级） |
| 缓存 | 难 | hash 文件名让缓存策略自动生效 |

## 🏋️ 小练习

1. 跑 `npm run build`，看默认产物（一个 JS chunk）；加 `manualChunks` 后再 build，对比 chunk 数量变化；
2. 用 `npm run preview` 本地预览 dist，确认线上效果正常。
