# 第 0 章 · 为什么是 Vite：裸 HTML 撞墙记

你可能习惯了这样的开发方式：新建一个 `index.html`，往里塞 `<script src="cdn/vue.js">`，再塞一堆业务代码，双击用浏览器打开，改完刷新。简单、直接、不用装任何东西。

这套方式写小东西很爽。但项目稍微一长大，你一定会撞墙。这一章不讲代码，只讲一件事：**为什么现代前端离不开构建工具，而那个工具为什么是 Vite。** 想通这一点，后面所有配置你都会有"原来如此"的感觉。

> 📌 本教程和主教程的[第 4 章](../../vue3/docs/04-从CDN到Vite.md)有部分重叠——那里是"够用就行"的工程化入门，这里会正面回答"Vite 凭什么这么快"。建议先读本章，再去动手。

---

## 一、裸 HTML / CDN 方式，撞的四面墙

用 `<script>` 标签 + CDN 写项目，等业务一复杂，你会撞上这四面墙：

**墙 1：所有代码挤在一个 html 里。** 几百上千行挤一团，找一个函数要 Ctrl+F 翻半天。想拆成多个 `.js` 文件？`<script src="a.js">` 能引入，但**文件之间的依赖顺序**全靠你脑子记——b.js 用了 a.js 的函数，a.js 就必须排在前面，顺序一错就 `xxx is not defined`。

**墙 2：用不了 `import`。** 现代的库（Vue、axios、Element Plus）都靠 `import { ref } from 'vue'` 这种 ES Module 语法引入。裸 html 里直接写 `import`，浏览器报错——因为 CDN 方式没有"模块解析"。于是你只能用全局变量 `window.Vue` 这种老办法，别扭。

**墙 3：上线没法优化。** 上线时代码要**压缩**（去掉空格注释）、**合并**（减少请求数）、**拆分**（按需加载）。这些靠手动做不现实，更没法做"tree-shaking"（删掉没用到的代码）。

**墙 4：改一行，整个页面刷新。** 改一个字，F5 整页重载，状态全丢，慢且打断思路。

jQuery 时代你也许用 RequireJS / sea.js 管过模块，也许用过 webpack。但那些要么过时、要么配置复杂到劝退。现代前端需要一个**又快又简单**的构建工具——这就是 **Vite**。

---

## 二、构建工具到底在干嘛

给裸 HTML 老哥一句话解释：**构建工具 = 帮你把"开发时方便人写"的代码，变成"浏览器能高效跑"的代码。**

它干这几件事：

| 你在开发时 | 构建工具帮你 |
|---|---|
| 用 `import` 分文件写，依赖随便引 | 解析依赖、按正确顺序拼接（不用你管 script 顺序） |
| 用 `.vue`、`.ts`、Sass、新语法 | 翻译成浏览器认识的 `.js` / `.css` |
| 写完点运行 | 起一个本地服务器，改代码**局部热更新**（不用整页刷新） |
| 准备上线 | 压缩、合并、拆 chunk、tree-shaking，输出到 `dist/` |

> 📌 一句话：**构建工具让你"用现代方式写代码"，它负责把现代代码翻译、组织、优化成浏览器能跑的东西。**

## 三、为什么是 Vite，不是 webpack

你一定听过 webpack。它曾经是前端构建的霸主。那凭什么现在大家都换 Vite？

答案藏在 Vite 的一个核心洞察里：**浏览器其实早就原生支持 ES Module 了。**

### webpack 的做法：先打包，再运行

webpack（包括早年的各种打包器）的思路是 **Bundle（打包）**：不管你开发还是上线，它都先把所有文件**搜罗、编译、拼成一个大 bundle**，再交给浏览器。项目一大，这个"打包"过程要几十秒甚至几分钟——**你按下启动，喝杯咖啡才 ready**。

### Vite 的做法（dev 时）：不打包，按需编译

Vite 的天才之处在于：**开发时根本不打包。** 它直接利用浏览器的原生 ESM——

- 你的 `index.html` 里写 `<script type="module" src="/src/main.js">`；
- `main.js` 里 `import { ref } from 'vue'`，浏览器看到 `import`，**自己**向 dev 服务器发请求要这个文件；
- Vite 服务器收到请求，**只编译当前这个文件**，原样返回；
- 浏览器再顺着 `import` 一层层按需请求，用到哪个编译哪个。

这就是 **No-Bundle（非打包）**：开发时按需编译、按需加载，启动**几乎瞬间**（不用等打包）。

> 🔑 **核心区别**：webpack 是"先打包好一坨，再启动"；Vite 是"你要啥我现编译啥，启动即用"。

### 上线时呢？

上线还是得打包（浏览器最终要的是优化过的文件）。Vite 上线用 **Rollup**（Vite 8 起默认用 Rust 重写的 **Rolldown**）打包，产出压缩、拆分的 `dist/`。这部分 Vite 帮你做，你几乎不用配置。

| | webpack | Vite |
|---|---|---|
| dev 启动 | 先打包，项目大则几十秒~分钟 | 不打包，**秒级** ready |
| dev 改代码 | 常要重新打包/整页刷新 | **HMR 局部热更新**，毫秒级 |
| 配置 | 复杂，劝退新手 | 极简，开箱即用 |
| 上线打包 | webpack 自己 | Rollup / Rolldown |
| Vue 官方推荐 | ❌（已不是首选） | ✅ |

入门直接用 Vite，没必要碰 webpack。等以后公司老项目用到，再补也不迟。

## 四、Vite 凭什么这么快（剧透第 6 章）

先剧透 Vite 快的三个原因，第 6 章会深入讲 HMR：

1. **No-Bundle**：dev 时不打包，启动即用（前面讲过）。
2. **esbuild 预构建**：Vite 用 Go 写的 `esbuild`（比 JS 写的工具快 10~100 倍）把 `node_modules` 里的依赖预先处理成浏览器好用的格式，这一步极快。
3. **HMR（热模块替换）**：你改一个 `.vue`，Vite 只把这个组件重新编译、**精确替换**到页面，**不刷新整页**——状态保留、毫秒生效。这就是你改代码后"瞬间看到效果"的秘密。

## 对照：裸 HTML/CDN 时代 vs Vite 工程化

| | 裸 HTML / CDN 时代 | Vite 工程化 |
|---|---|---|
| 引入库 | `<script src="cdn/vue.js">` | `npm i vue` + `import { ref } from 'vue'` |
| 多文件组织 | 全塞 index.html，script 顺序靠脑子记 | `src/` + `import` + 别名 `@`，依赖自动解析 |
| 改代码看效果 | 手动 F5 整页刷新，状态丢失 | HMR 秒级局部热更，状态保留 |
| 用新语法 / `.vue` / Sass | 用不了 | 自动编译翻译 |
| 上线 | 手动处理，难以优化 | `vite build` 压缩拆 chunk 到 `dist/` |

省掉的，是**手管 script 顺序、手动刷新、手动优化**这些琐碎活；换来的，是**专注写业务、改完即见、上线一键**的顺畅体验。
