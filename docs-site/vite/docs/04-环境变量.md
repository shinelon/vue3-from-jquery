# 第 4 章 · 环境变量：dev 和线上用不同配置

你的笔记应用，开发时连本地的 mock 接口（`localhost:3000`），上线后要连真实的服务器地址。这俩地址不一样，你总不能每次上线手动改代码吧？环境变量就是解决这个的——**让同一份代码，在不同环境跑出不同配置**。

> 📂 配套项目 `project/vite-notes/`，本章对应分支 `vite/03-env`。

---

## 一、痛点：硬编码的配置

裸 HTML 时代，你可能这么干：

```js
const API_BASE = 'http://localhost:3000'   // 开发时
// 上线前，手动改成：
// const API_BASE = 'https://api.mysite.com'
```

问题：
1. **容易忘改**——上线忘改地址，请求打到本地，线上挂掉；
2. **多人协作乱**——你的本地地址和别人不一样，老冲突；
3. **敏感信息**——API key、token 这类，硬编码进代码还容易泄露。

现代做法：用**环境变量**，把"随环境变的配置"抽到代码外。

---

## 二、Vite 的环境变量：`.env` 文件 + `import.meta.env`

Vite 内置了环境变量支持，零配置。在项目根建 `.env` 文件：

```bash
# .env —— 所有环境都加载的默认值
VITE_APP_TITLE=vite-notes
```

> ⚠️ **关键规则**：只有以 **`VITE_`** 开头的变量，才会暴露给你的代码。这是 Vite 的安全设计——防止你不小心把数据库密码之类也塞进前端。

代码里用 `import.meta.env` 读取：

```js
const title = import.meta.env.VITE_APP_TITLE   // 'vite-notes'
```

> 📌 `import.meta.env` 是 Vite 注入的全局对象，dev 和 build 时都会把 `.env` 的值替换进去。

---

## 三、按环境区分：`.env.production` 等

光一个 `.env` 不够。Vite 按 **mode（模式）** 加载不同文件：

| 文件 | 何时加载 |
|---|---|
| `.env` | 所有模式都加载（默认值） |
| `.env.local` | 所有模式，但**被 git 忽略**（放本地私有配置） |
| `.env.development` | `npm run dev` 时（mode=development） |
| `.env.production` | `npm run build` 时（mode=production） |

**优先级**：`.env.production` > `.env`（同名变量，production 的覆盖默认）。

举个例子，给笔记应用配不同环境的 API 地址：

```bash
# .env —— 默认
VITE_APP_TITLE=vite-notes
VITE_API_BASE=/api
```

```bash
# .env.development —— 开发专用
VITE_API_BASE=http://localhost:3000
```

这样：开发时 `import.meta.env.VITE_API_BASE` 是 `http://localhost:3000`，打包上线时是 `/api`。**同一份代码，两个环境，自动切换**。

---

## 四、什么是 mode（模式）

Vite 有三个内置 mode：
- **`development`**：`npm run dev` 默认；
- **`production`**：`npm run build` 默认；
- **`test`**：测试时（本教程不涉及）。

你也可以自定义 mode，比如预发布环境：

```bash
vite build --mode staging    # 会加载 .env.staging
```

在 `package.json` 加个脚本：

```json
"scripts": {
  "build:staging": "vite build --mode staging"
}
```

## 六、`.env` 该不该提交 git？

| 文件 | 提交吗 | 说明 |
|---|---|---|
| `.env` | ✅ 提交 | 默认配置，团队共享 |
| `.env.production` | ✅ 提交 | 生产配置（非敏感） |
| `.env.local` | ❌ 不提交 | 本地私有/敏感（Vite 默认忽略它） |

> 🔑 **铁律**：敏感信息（API key、密码）放 `.env.local`（不提交），或放后端由服务端注入，**绝不**放会被打包进前端的 `VITE_` 变量里——前端代码用户能看到，等于公开。

---

## 对照：配置管理这件事

| | 裸 HTML 时代 | Vite 环境变量 |
|---|---|---|
| 环境差异 | 手动改代码 | `.env.development` / `.env.production` 自动切 |
| 敏感信息 | 硬编码易泄露 | `.env.local` 不提交 |
| 暴露给前端 | — | 只暴露 `VITE_` 前缀的 |
